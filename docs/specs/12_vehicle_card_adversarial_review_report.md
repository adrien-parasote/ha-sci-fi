## Part 3: Execution Simulation Walkthrough

We simulated a complex multi-vehicle setup:
1. **Initial Mount:** Two vehicles are configured. Vehicle A (`_active_vehicle_id = 0`) is displayed. The temperature wheel defaults to index 2 (18°C).
2. **User Interaction (AC Temp Change):** The user changes the temperature wheel to 22°C (`this._selected_temp_id = 6`).
3. **User Navigation:** The user clicks "Next" (`_next()`), setting `_active_vehicle_id = 1` (Vehicle B).
4. **Resulting State:** The temperature wheel *remains* at 22°C because `_selected_temp_id` is a component-wide reactive state, not vehicle-specific. If the user clicks "Start AC" on Vehicle B, it sends a 22°C request.
5. **Issue Discovered:** There is no loading indicator or button-disabling state while the async service call `hass.callService()` is executing. If a service call takes 2-3 seconds, the user might double-click or assume the card has crashed.

## Part 5: Detailed Adversarial Review Findings

### 5.1 [CRITICAL] — Hand-Modifying `fr.ts` Overwrite Risk
- **Location:** `§ Localization — New msg() Keys`
- **Problem:** The spec states that new translation keys "must be added to `src/locales/locales/fr.ts`". However, `fr.ts` is auto-generated by the `lit-localize build` command. Direct edits to this file are explicitly forbidden and will be wiped out upon regeneration.
- **Fix:** Rewrite the paragraph to clarify that translations must only be added to `xliff/fr.xlf` using the standard extraction workflow, and `fr.ts` is compiled from it:
  ```markdown
  > [!IMPORTANT]
  > Do NOT modify `src/locales/locales/fr.ts` by hand. New translation keys must be added to `xliff/fr.xlf`, then compiled via `npm run locale:build` to regenerate `fr.ts` (relying on `lit-localize` hash matching).
  ```

### 5.2 [HIGH] — Broken `_temperature_items` Class-Property Initialization
- **Location:** `§ Reactive state` vs `§ _renderActions()`
- **Problem:** The spec declares `private readonly _temperature_items` as a class property referencing `this.hass?.config?.unit_system?.temperature`. During class instantiation, `this.hass` is guaranteed to be `undefined`, so it defaults to `'°C'` and will never reactively update if the Home Assistant temperature unit changes. Meanwhile, `_renderActions()` also redeclares `tempItems` locally, creating redundant and confusing code.
- **Fix:** Completely delete the class property '_temperature_items' from 'sci-fi-vehicles.ts' specifications, and rely exclusively on the dynamically built `tempItems` inside `_renderActions()`.

### 5.3 [HIGH] — Lack of Loading/Disabled States for AC Actions
- **Location:** `§ _renderActions()` and `_startAc() / _stopAc()`
- **Problem:** Making a service call is an asynchronous network request. The spec lacks any loading state property (e.g., `_is_ac_loading: boolean`), meaning buttons remain clickable and provide no feedback until the toast pops up.
- **Fix:** Add a `@state() private _is_ac_loading: boolean = false;` reactive property. Disable both AC buttons (`.disabled` or property binding) while `_is_ac_loading` is `true`. Toggle it in the promise lifecycle:
  ```ts
  private _startAc(v: SciFiVehicleEntry): void {
    this._is_ac_loading = true;
    const temperature = this._selected_temp_id + 16;
    void this.hass.callService(HASS_RENAULT_SERVICE, HASS_RENAULT_SERVICE_ACTION_START_AC, {
      vehicle: v.id,
      temperature,
    })
    .then(() => this._toast(false, msg('done')))
    .catch((e: Error) => this._toast(true, e.message))
    .finally(() => { this._is_ac_loading = false; });
  }
  ```

### 5.4 [MEDIUM] — Fallback for Missing Autonomy and Mileage Units
- **Location:** `§ Derived values`
- **Problem:** If a sensor entity has no `unit_of_measurement` attribute, formatting with `+ ' ' + unit` will render `undefined` or a blank string.
- **Fix:** Explicitly specify fallbacks for units:
  - Mileage fallback: `'km'`
  - Autonomy fallback: `'km'`
  - Fuel quantity fallback: `'%'` (percentage remaining)

### 5.5 [MEDIUM] — GPS Navigation Device Detection Logic
- **Location:** `§ _renderTop() zone`
- **Problem:** The spec mentions that the button "opens Google Maps / Apple Maps based on device", but doesn't define the template or device check.
- **Fix:** Document the exact helper check for iOS/macOS:
  ```ts
  const isApple = /iPad|iPhone|iPod|Macintosh/.test(navigator.userAgent);
  const url = isApple 
    ? `maps://?q=${latitude},${longitude}` 
    : `https://maps.google.com/?q=${latitude},${longitude}`;
  window.open(url, '_blank');
  ```
